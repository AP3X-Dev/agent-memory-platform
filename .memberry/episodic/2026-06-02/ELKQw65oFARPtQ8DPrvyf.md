[project:amp] Root-caused and fixed the long-standing "typedHandler is not a function" error that broke amp_store and every other mutating tool while reads worked. Cause was NOT the connection or a server restart — it was tool registration. Mutating tools were registered as server.tool(name, desc, Schema, {}, handler) with an EMPTY {} as the ToolAnnotations arg. The MCP SDK overload parser (isZodRawShapeCompat) treats {} as a zero-param Zod raw shape, so it never consumes {} as annotations; the handler shifts into the annotations slot and the callback slot becomes the {} object — not callable. Read tools worked because they passed a non-empty annotations object (ANN_READONLY_IDEMPOTENT). Fix: never pass empty {} as annotations. Added ANN_WRITE = { readOnlyHint: false } in @amp/mcp tools.ts and replaced all 6 empty-{} registrations (amp_store, amp_memory_insert, amp_memory_replace, amp_memory_rewrite, amp_memory_promote, amp_consolidate). Found and fixed the full error class across packages too: arch (2), research (3), wiki (1) used "{} satisfies ToolAnnotations" — replaced with "{ readOnlyHint: false } satisfies ToolAnnotations". 12 registrations total. Added tool-registration.regression.test.ts that registers tools on a real McpServer and asserts every wired _registeredTools[name].handler is a function (the prior tools.test.ts bypassed server.tool() via buildToolHandlers, which is why it never caught this). Verified the guard fails when the bug is reintroduced. Full build clean; 534 tests pass. Restarted amp-mcp.service (runs tsx on source). This store call itself is the end-to-end confirmation the write path works again. This supersedes the earlier belief recorded in memory that amp_store failures were a flaky write handler / restart issue — the real cause was static and now fixed.
